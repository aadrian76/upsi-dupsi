# Capítulo 3: Evaluación de Madurez Cloud

## ✅ Madurez en la Nube
Un modelo de madurez de la nube es un marco estructurado que ayuda a las organizaciones a evaluar y mejorar sus capacidades en la nube a través de etapas evolutivas definidas. Estos modelos ofrecen enfoques escalonados para proteger entornos de nube pública, privada o híbrida, estableciendo parámetros de progreso claros. Sirve para:
*   **Evaluar el estado actual** de adopción y competencias cloud.
    
*   **Identificar brechas y riesgos** en tecnología, seguridad y gobernanza.
    
*   **Definir una hoja de ruta clara** con pasos para avanzar.
    
*   Alinear la adopción con los objetivos de negocio, no solo tecnológicos.

## ✅ Niveles de Madurez
Un ejemplo de como clasificar los niveles de madurez puede ser:
*   **Pendiente**  
    La organización tiene poco o ningún uso de servicios cloud, no hay procesos definidos y la nube se emplea de forma experimental.
    
*   **Inicial**  
    Se empieza a usar infraestructura como servicio (_IaaS_), pero aún con prácticas informales y sin estandarización.
    
*   **Gestionado**  
    Se establecen procesos básicos repetibles para migrar y gestionar recursos y se documentan las prácticas principales.
    
*   **Definido**  
    La adopción se extiende, hay políticas, gobernanza y herramientas que permiten un control y seguimiento más consistentes.
    
*   **Optimizado**  
    La nube es parte integral de la estrategia, con automatización, seguridad continua, eficiencia de costes y mejora constante.

## ✅ Modelos de evaluación de madurez

### OWASP SAMM
El modelo OWASP SAMM (Software Assurance Maturity Model) es un marco estructurado que sirve para evalúar la madurez de procesos de seguridad en el desarrollo de software. Su objetivo es identificar áreas de riesgo en el ciclo de vida del software y establecer prácticas que permitan prevenir vulnerabilidades desde las etapas iniciales del desarrollo. A través de evaluaciones periódicas y métricas definidas, las empresas pueden implementar acciones correctivas, reforzar controles y promover la cultura de seguridad entre los equipos de desarrollo, asegurando que las aplicaciones se construyan de manera confiable y segura.

### BSIMM

El modelo BSIMM (Building Security In Maturity Model) es un marco de referencia que permite a las organizaciones evaluar cómo integran la seguridad dentro del ciclo de vida del desarrollo de software. Su finalidad es identificar prácticas efectivas y compararlas con las de otras empresas, para fortalecer la gestión de riesgos y la protección de las aplicaciones. A través de la observación de actividades concretas y la medición de su madurez, las organizaciones pueden diseñar estrategias de seguridad más precisas, implementar controles adecuados y promover una cultura de desarrollo seguro en toda la empresa.

### PESA
El Plan Estratégico de Seguridad en Aplicaciones (PESA) es un documento que define los criterios para evaluar y mejorar el nivel de madurez de las actividades de seguridad en aplicaciones dentro del Ciclo de Desarrollo Seguro del Software (S-SDLC) dentro de una organización.
  
Las actividades y sus criterios se organizan utilizando la metodología Accenture DNA, el estándar desarrollado por el equipo de Application Security para implantar un S-SDLC. Esta metodología está alineada con modelos de referencia como OWASP SAMM y BSIMM.

El DNA se organiza en 22 capacidades que abarcan los diferentes aspectos de la seguridad en aplicaciones a lo largo de su desarrollo. Estas capacidades se agrupan en 7 dominios: Security Governance, Security Training, Secure Design, Secure Implementation, Security Validation, Secure Orchestration y Security Operations.
